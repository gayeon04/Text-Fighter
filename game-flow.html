Strategy 패턴 — BattleManager

전투 알고리즘을 인터페이스로 분리하여 런타임에 교체 가능하도록 설계.

classDiagram class BattleStrategy { <<interface>> +execute() void } class NormalBattle { +execute() void } class BossBattle { +execute() void } class BattleManager { -BattleStrategy strategy +setStrategy(BattleStrategy) void +startBattle() void } BattleStrategy <|.. NormalBattle BattleStrategy <|.. BossBattle BattleManager o-- BattleStrategy
전투 알고리즘 캡슐화 런타임 전략 교체 가능 OCP 원칙 준수

Observer 패턴 — QuestManager

게임 이벤트(처치, 코인 획득 등) 발생 시 자동으로 퀘스트 진행도를 업데이트.

classDiagram class GameEventListener { <<interface>> +onEvent(String type, int value) void } class QuestManager { -List~Quest~ quests +onEvent(String type, int value) void +checkQuests() void +getInstance() QuestManager } class Quest { -String type -int goal -int progress +update(int value) void +isComplete() boolean } class Game { -List~GameEventListener~ listeners +notifyListeners(String, int) void } GameEventListener <|.. QuestManager QuestManager "1" *-- "many" Quest Game --> GameEventListener : notify
느슨한 결합(Loose Coupling) 이벤트 기반 업데이트 퀘스트 자동 추적

Singleton 패턴 — GameLogger

게임 전체에서 로그 인스턴스를 하나만 유지.

classDiagram class GameLogger { -static GameLogger instance -GameLogger() +getInstance() GameLogger +log(String message) void +warn(String message) void +error(String message) void } GameLogger --> GameLogger : instance
단일 인스턴스 보장 전역 접근점

Factory 패턴 — EnemyRegistry

적 객체 생성 로직을 중앙화하여 코드 중복 제거.

classDiagram class EnemyRegistry { +create(String type) Enemy } class Enemy { -String name -int hp -int damage +getName() String +getHp() int } class SlimeEnemy { +SlimeEnemy() } class DragonEnemy { +DragonEnemy() } EnemyRegistry ..> Enemy : creates Enemy <|-- SlimeEnemy Enemy <|-- DragonEnemy
생성 로직 캡슐화 새 적 추가 용이

멀티스레드 — AutoSaveTask

30초마다 백그라운드에서 자동 저장을 수행하는 스레드 구조.

sequenceDiagram participant Main as Game (메인 스레드) participant Task as AutoSaveTask (백그라운드) participant File as 저장 파일 Main->>Task: new Thread(AutoSaveTask).start() loop 30초마다 Task->>Task: Thread.sleep(30000) Task->>File: saveGame() Task->>Main: GameLogger.log("자동 저장 완료") end Main->>Task: interrupt() [게임 종료 시]
Runnable 구현 비동기 저장 메인 스레드 블로킹 없음 인터럽트 안전 종료

적용된 디자인 패턴 요약

이번 구현에서 적용한 4가지 디자인 패턴과 역할.

패턴 적용 위치 목적 효과
Strategy BattleManager 전투 알고리즘 교체 일반전 / 보스전 전략 분리, 확장 용이
Observer QuestManager 이벤트 기반 퀘스트 업데이트 게임 이벤트와 퀘스트 로직 분리
Singleton GameLogger 전역 로거 단일화 일관된 로그 관리, 중복 인스턴스 방지
Factory EnemyRegistry 적 객체 생성 중앙화 생성 로직 캡슐화, 새 적 추가 용이
Multithreading AutoSaveTask 백그라운드 자동 저장 30초 주기 저장, 메인 스레드 비차단

개선 전 / 후 비교

UI 및 구조 개선 사항.

개선 전 (Before)

  • UI 텍스트 전체 영어
  • 구분선 길이 불일치 (39자 ~ 67자)
  • 섹션 구분 없는 정보 나열
  • 전투 알고리즘이 Game 클래스에 직접 작성
  • 퀘스트 진행도 수동 체크
  • 로거 인스턴스 여러 개 생성 가능
  • 자동 저장 없음

개선 후 (After)

  • UI 전체 한국어화
  • 구분선 60자로 표준화
  • [섹션명] 헤더로 가독성 향상
  • BattleManager(Strategy)로 전투 분리
  • QuestManager(Observer)로 자동 퀘스트 추적
  • GameLogger(Singleton)로 단일 로거 보장
  • AutoSaveTask(Thread)로 30초 자동 저장
